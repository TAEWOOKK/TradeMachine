## 3. 기간별 시세 API (`FHKST03010100`) — MA 계산용 검증

> **출처: 공식 GitHub `inquire_daily_itemchartprice.py`, `chk_inquire_daily_itemchartprice.py`**

### 기본 확인

| 항목 | 검증 결과 |
|------|----------|
| Endpoint | `GET /uapi/domestic-stock/v1/quotations/inquire-daily-itemchartprice` — **존재** |
| TR ID | `FHKST03010100` — **확인** |
| 최대 조회 건수 | **한 번에 최대 100건** (100일치 일봉) |
| 기간 분류 | `D`(일봉), `W`(주봉), `M`(월봉), `Y`(년봉) — **모두 지원** |
| 수정주가 | `fid_org_adj_prc=0` 으로 수정주가 조회 가능 — **지원** |

### MA 계산 가능성

| 이동평균선 | 필요 일수 | 100건 제한 내? | 판정 |
|-----------|----------|--------------|------|
| MA(5) | 5일 | O | **가능** |
| MA(20) | 20일 | O | **가능** |
| MA(60) | 60일 | O | **가능** |
| MA(120) | 120일 | **X (100건 초과)** | 2번 호출로 해결 가능 |

> MA(60)까지는 한 번 호출로 가능. MA(120) 이상이 필요하면 날짜 범위를 나눠 2회 호출.
> 우리 계획의 MA(5), MA(20)은 **한 번 호출로 충분**.

### 일봉 응답 필드 (output2 배열)

| API 필드 | 한글명 | MA 계산 사용 |
|---------|--------|------------|
| `stck_bsop_date` | 영업 일자 | 날짜 기준 |
| `stck_clpr` | **종가** | **MA 계산에 사용** |
| `stck_oprc` | 시가 | 참고 |
| `stck_hgpr` | 고가 | 참고 |
| `stck_lwpr` | 저가 | 참고 |
| `acml_vol` | 거래량 | 거래량 분석 |
| `acml_tr_pbmn` | 거래대금 | 거래대금 분석 |
| `flng_cls_code` | **락 구분 코드** | **배당락/권리락 감지 가능!** |
| `prtt_rate` | **분할 비율** | **주식 분할 감지 가능!** |
| `mod_yn` | 변경 여부 | 데이터 수정 여부 확인 |

> **핵심 발견:**
> - `flng_cls_code`로 **배당락을 자동 감지**할 수 있다 → 배당락 가짜 신호 필터링 가능
> - `prtt_rate`로 **주식 분할을 감지**할 수 있다
> - `fid_org_adj_prc=0`으로 **수정주가**를 조회하면 분할/병합 시 MA 왜곡 자동 방지

## 7. 계획 변경 사항 (검증으로 인한 개선)

### 7.1 시가총액 필터 — Phase 5 → Phase 2로 앞당김

**이전 계획:** 시가총액 필터는 별도 종목정보 API가 필요하므로 Phase 5
**검증 결과:** 현재가 시세 API(`inquire-price`)에 `hts_avls`(시가총액) 필드가 포함
**변경:** 초기부터 시가총액 필터 적용 가능. 추가 API 호출 불필요.

### 7.2 관리종목/투자유의/거래정지 필터 — 신규 추가

**이전 계획:** 거래정지는 "비정상 응답으로 감지" (불안정한 방식)
**검증 결과:** 현재가 시세 API에 명시적 필드 존재
- `mang_issu_cls_code`: 관리종목 여부
- `invt_caful_yn`: 투자유의 여부
- `temp_stop_yn`: 임시 정지 여부
- `sltr_yn`: 정리매매 여부

**변경:** 명시적 필드로 안전하게 필터링 가능. 종목 품질 필터에 추가.

### 7.3 배당락 감지 — Phase 5 → Phase 3으로 앞당김

**이전 계획:** 배당락일 캘린더 데이터 별도 연동 필요 (Phase 5)
**검증 결과:** 일봉 데이터(`inquire-daily-itemchartprice`)의 `flng_cls_code`로 배당락 감지 가능
**변경:** 일봉 데이터에서 자동 감지하여 MA 판단 시 참고 가능.

### 7.4 수정주가로 MA 계산 — 확인

**이전 계획:** 주식 분할 시 MA 왜곡 대응으로 변동률 30% 초과 필터
**검증 결과:** `fid_org_adj_prc=0` 파라미터로 수정주가 조회 지원
**변경:** 수정주가 기반으로 MA 계산하면 분할/병합에 의한 왜곡 자체가 발생하지 않음.

## 8. 주의 사항 (검증 중 발견)

### 8.1 일봉 데이터 100건 제한

- 한 번 호출에 최대 100건
- MA(20) 계산에는 충분하지만, 향후 MA(120) 등 장기 이동평균선 필요 시 2회 호출 필요
- 날짜 범위를 나눠 호출하면 해결 가능

### 8.2 모든 숫자값이 문자열로 반환

- KIS API는 숫자 필드도 `"70000"` 같은 **문자열**로 반환
- Repository 계층에서 int/float 변환 처리 필요
- 주문 시에도 수량(`"10"`), 가격(`"70000"`)을 문자열로 보내야 함

### 8.3 주문 Body도 문자열

- `ORD_QTY: "10"` (숫자 10이 아닌 문자열 "10")
- `ORD_UNPR: "70000"` (숫자가 아닌 문자열)
- 시장가 주문 시 `ORD_UNPR: "0"`

## 9. 최종 API 호출 횟수 산정

### 1회 스캔 (5분 간격) 기준

```
관심 종목 5개, 보유 종목 최대 5개 가정:

1. 잔고 조회                          = 1건
2. 미체결 주문 조회                     = 1건
3. 보유 종목 현재가 조회 (5종목)         = 5건
4. 관심 종목 현재가 조회 (5종목)         = 5건
5. 관심 종목 일봉 조회 (5종목, MA 계산)  = 5건
6. 매수가능금액 조회 (매수 시)           = 0~1건
7. 주문 실행 (매수/매도)                = 0~2건
────────────────────────────────────────
1회 스캔 합계: 약 17~19건

초당 환산: 19건 / 300초(5분) = 0.06건/초
호출 제한 (초당 20건) 대비: 여유 충분
```

## 10. KIS API 호출 제한 상세 분석

> 조사일: 2026-03-11
> 소스: KIS Developers 포털, 공식 GitHub, 개발자 블로그, 실사용자 경험

### 10.1 KIS API 공식 호출 제한 정책

| 제한 유형 | 수치 | 근거 | 확인 수준 |
|-----------|------|------|-----------|
| **초당 REST API 호출** | **초당 20건** | 공식 문서, 다수 개발자 확인 | **공식 확인** |
| **토큰 발급 빈도** | **1분에 1회** | 초과 시 `EGW00133` 에러 반환 | **공식 확인** |
| **일일 호출 총량** | **구체적 수치 미공개** | "하루 총 호출 횟수 정해져 있음"만 안내 | **존재하나 수치 미공개** |
| **월별 호출 총량** | **구체적 수치 미공개** | 공식 문서에 언급 없음 | **확인 불가** |
| **동시 접속** | **1 세션** | 동일 계정 다중 동시 호출 제한 | **공식 확인** |
| **WebSocket 구독** | **세션당 41종목** | 공식 안내 | **공식 확인** |

#### 초과 시 벌칙

| 상황 | 결과 |
|------|------|
| 초당 20건 초과 | HTTP 429 "Too Many Requests" 에러 |
| 429 연속 발생 | **1분~10분 일시적 API 차단** |
| 심한 경우 | **계정 일시 정지** |
| 토큰 1분 내 재발급 | `EGW00133` 에러 (치명적이지 않음, 기존 토큰 유효) |

#### 비공개 제한에 대한 참고 사항

### 10.3 제한 초과 여부 판정

#### 초당 제한 (초당 20건) — 핵심 제한

```
[평균]
  일반 시나리오: 1,359건 / 23,400초(6.5시간) = 초당 0.06건 → 매우 여유

[순간 최대 (1회 스캔 내 연속 호출)]
  최악: 28건의 API를 약 3~5초 내에 호출
  → 초당 5.6~9.3건 → 초당 20건 미만 ✓

  하지만 httpx 비동기 호출로 동시에 보내면:
  → 5개 종목 현재가를 async로 동시 호출 → 순간 5건/초 → 안전
  → 10개 종목 (보유5 + 관심5) 동시 호출 → 순간 10건/초 → 안전

  ★ 판정: 초당 20건 초과하지 않음. 여유 있음.
  ★ 단, 안전을 위해 초당 10건 이하로 보수적 설정 권장.
```

#### 일일 제한 (구체적 수치 미공개)

```
  실사용자 보수적 설정: 일일 5,000건
  우리 일반 시나리오: ~1,359건
  우리 최대 시나리오: ~2,190건

  ★ 판정: 5,000건 기준 시 일일 제한의 27~44% 수준. 안전.
  ★ 단, 공식 수치가 미공개이므로 모니터링 필요.
```

#### 월별 제한 (확인 불가)

```
  공식 월별 제한 수치 없음.
  일일 제한을 넘지 않으면 월별도 문제 없을 것으로 판단.

  ★ 판정: 확인 불가하나, 일일 기준 여유 있으므로 문제 없을 것으로 추정.
```

#### 토큰 발급 (1분에 1회)

```
  우리 프로그램: 08:50에 1회 발급 후 캐싱.
  만료 임박 시 1회 갱신 (하루 최대 2회).
  스캔 중 만료 시 재발급 1회.

  ★ 판정: 전혀 문제 없음.
```

### 10.5 안전 강화를 위한 권장 사항

```
[필수 (Phase 2)]
  1. Rate Limiter 구현: 초당 최대 10건으로 보수적 설정
  2. API 호출 카운터: 일일 호출 횟수를 메모리에서 추적
  3. 429 에러 핸들링: 지수 백오프 (1초 → 2초 → 4초)

[권장 (Phase 3~4)]
  4. 일봉 캐싱: 같은 스캔 내 보유종목 일봉과 관심종목 일봉이 겹치면 재사용
  5. API 응답 캐싱: 현재가 데이터를 짧은 TTL(5초)로 캐싱
  6. 호출 모니터링 로그: 매 스캔의 API 호출 건수 기록
  7. 일일 한도 경고: 추정 일일 한도(5,000건)의 80% 도달 시 경고 로그

[추후 (Phase 5)]
  8. KIS 포털에서 정확한 일일/월별 제한 수치 확인 후 설정 반영
  9. 토큰 로테이션 (다중 계좌 운영 시)
```
